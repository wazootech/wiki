# Wiki Plan Guide

Every implementation plan is written for an executor model that has **zero context**: it has not seen the advisor session, the audit, or any prior conversation. It should be written so that it can be executed with maximum reliability.

Three properties make a plan executable by a weaker model:

1. **Self-contained context** — everything needed is in the file: paths, configuration snippets, and page frontmatter.
1. **Verification gates** — every step ends with a command and its expected result. The executor never has to guess if a step succeeded.
1. **Hard boundaries and escape hatches** — explicit out-of-scope list and "STOP and report" conditions.

File naming: `plans/NNN-short-slug.md`, numbered in recommended execution order.

## Template

```markdown
# Plan NNN: <Imperative title — what will be true after this plan>

> **Executor instructions**: Follow this plan step by step. Run every
> verification command and confirm the expected result before moving to the
> next step. If anything in the "STOP conditions" section occurs, stop and
> report — do not improvise. When done, update the status row for this plan
> in `plans/README.md` — unless a reviewer dispatched you and told you they
> maintain the index.
>
> **Drift check (run first)**: `git diff --stat <planned-at SHA>..HEAD -- <in-scope paths>`
> If any in-scope file changed since this plan was written, compare the
> "Current state" excerpts against the live code/markdown before proceeding; on a
> mismatch, treat it as a STOP condition.

## Status

- **Priority**: P1 | P2 | P3
- **Effort**: S | M | L
- **Risk**: LOW | MED | HIGH
- **Depends on**: plans/NNN-*.md (or "none")
- **Category**: integrity | conventions | formatting | config | deploy | docs | style
- **Planned at**: commit `<short SHA>`, <YYYY-MM-DD>
- **Issue**: <GitHub issue URL — only when published via `--issues`; omit otherwise>

## Why this matters

2–5 sentences. The problem, its concrete cost, and what improves when this
lands. Written so the executor (and a human reviewer) understands the intent.

## Current state

The facts the executor needs, inlined:

- The relevant files, each with one line on its role:
  - `wiki/Opal_Security.md` — page with broken link to `Gregory_Davidson.md`
- Excerpts of the code/markdown as it exists today (short, with line markers).
- The wiki conventions that apply here:
  "Link style is standard page links — see `wiki.yml:link.style: standard`. Internal links must use `[text](Page_Name.md)` format."

## Commands you will need

| Purpose   | Command                  | Expected on success |
|-----------|--------------------------|---------------------|
| Format    | `wiki -c <config> fmt`   | exit 0              |
| Lint      | `wiki -c <config> lint`  | exit 0, no warnings |
| Integrity | `wiki -c <config> check` | exit 0, no errors   |
| Render    | `wiki -c <config> render`| exit 0              |

(Exact commands from this repository — using the resolved config file `-c docs/wiki.yml` or similar.)

## Scope

**In scope** (the only files you should modify):
- `wiki/Opal_Security.md`
- `wiki.yml`

**Out of scope** (do NOT touch):
- `skills/` folder
- Render output in `_site/` (this is automatically generated by build)

## Steps

### Step 1: <imperative title, e.g., Repair broken links in Opal_Security.md>

What to do, precisely. Specify exact lines and the target content shape.

**Verify**: `wiki -c <config> lint` → exit 0, or specifically confirm the error on `Opal_Security.md` is gone.

### Step 2: ...

## Test plan

- Run full audit validation:
  `wiki -c docs/wiki.yml fmt --check && wiki -c docs/wiki.yml lint --strict && wiki -c docs/wiki.yml check --strict`
- Verification: all steps exit 0 with no errors.

## Done criteria

Machine-checkable. ALL must hold:

- [ ] `wiki -c <config> fmt --check` exits 0
- [ ] `wiki -c <config> lint --strict` exits 0
- [ ] `wiki -c <config> check --strict` exits 0
- [ ] No files outside the in-scope list are modified (`git status`)
- [ ] `plans/README.md` status row updated

## Stop conditions

Stop and report back (do not improvise) if:

- The markdown or config at the locations in "Current state" doesn't match the excerpts.
- A step's verification fails twice after a reasonable fix attempt.
- The fix appears to require touching an out-of-scope file.

## Maintenance notes

- Any follow-up or related layout/formatting changes deferred out of this plan.
```

## Index file (plans/README.md)

Written once by the advisor after all plans are written:

```markdown
# Wiki Implementation Plans

Generated by the wiki skill on <date>. Execute in the order below. Each executor: read the plan fully before starting, honor its STOP conditions, and update your row when done.

## Execution order & status

| Plan | Title | Priority | Effort | Depends on | Status |
|------|-------|----------|--------|------------|--------|
| 001  | ...   | P1       | S      | —          | TODO   |
| 002  | ...   | P1       | M      | 001        | TODO   |

Status values: TODO | IN PROGRESS | DONE | BLOCKED | REJECTED

## Dependency notes

- 002 requires 001 because <reason>.

## Findings considered and rejected

- <finding>: not worth doing because <one line>.
```
